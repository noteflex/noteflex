# 문서 자동 갱신 원칙

> **목적**: 사용자가 매 세션마다 "문서에 적용해줘"라고 요청하지 않아도 Claude Code가 자동으로 3개 문서를 최신 상태로 유지한다.
> **적용 시점**: 작업 완료 직후 (commit 직전 또는 직후)

---

## 자동 갱신 대상 3개 파일

| 파일 | 갱신 시점 | 기록할 내용 |
|---|---|---|
| `docs/archive/sessions/noteflex-session-log.md` | 모든 sprint 완료 후 | 날짜·sprint 제목·commit hash·변경 파일·검증 결과·후속 영역 |
| `docs/active/PENDING_BACKLOG.md` | 출시 전·후 영향 항목 발견 시 | 출시 전 필수 vs 출시 후 즉시 분기, 우선순위·시간 추정 |
| `docs/active/BLOG_LOG.md` | 블로그 작업(글 수·정책·DOI) 변동 시 | 정책 변경·글 수 증감·이미지 정책·DOI 검증 결과 |

---

## 세션 로그 갱신 규칙 (`noteflex-session-log.md`)

- **위치**: 파일 상단 (시간 역순 — 최신이 위)
- **형식**:
  ```markdown
  ## YYYY-MM-DD — [sprint 제목]
  
  ### 완료
  - ✅ **작업명** (`커밋해시`) — 변경 파일·핵심 내용
  
  ### 짚힌 영역
  - ⚠️ 후속 확인 필요한 영역
  
  ### 결정 적용됨
  - 기술적·제품적 결정 사항
  
  ### 검증
  - X/X 테스트 PASS, 빌드 에러 X
  ```
- 하나의 날짜에 여러 sprint = 한 섹션에 합침 (섹션별 `####` 소제목)

---

## PENDING_BACKLOG 갱신 규칙

- **위치**: 파일 상단의 최신 날짜 섹션 (`## YYYY-MM-DD 추가 PENDING`)
- **완료 항목 처리**: 항목 앞에 `✅` 추가 또는 항목 제거 (중요 결정은 이력 보존)
- **새 항목 추가**: `출시 전 필수` vs `출시 후 즉시` 구분 명확히
- **박지 말 것**: 실험·임시 영역, 사용자 미확정 영역

---

## 블로그 로그 갱신 규칙 (`docs/active/BLOG_LOG.md`)

- 글 수 변동 시: 누적 글 수 표 갱신
- 이미지·DOI 정책 변경 시: 정책 결정 섹션 추가/수정
- DOI 검증 결과: 날짜별로 표에 추가
- 작업 이력: 커밋 단위로 표에 추가

---

## 박지 말 것 (exclusions)

- 실험·임시 작업 (사용자 확정 전)
- 코드 패턴·컨벤션·파일 구조 (코드 읽으면 알 수 있는 것)
- git 이력으로 알 수 있는 것 (`git log`, `git blame`)
- 이미 `CLAUDE.md` 또는 다른 공식 문서에 적용된 것
