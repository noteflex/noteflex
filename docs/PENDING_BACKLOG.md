## 0. 사용자 검증으로 확인된 버그 🔴⭐

다음 세션에서 가장 먼저 잡을 항목.

### 0.1 N+2 재출제 즉시 등장 버그 🐛
**사용자 검증**: "첫 번째 음표를 2번 틀린 후 정답을 맞추면 바로 다음 음표가 같은 음표가 나온다."

- 사용자 의도: 같은 음표가 최대한 연속으로 안 나오게
- 실제 동작: 정답 후 같은 음표가 즉시 다음 자리에 등장
- 추정 원인: turn counter 증가 타이밍 + retry queue due 계산 어긋남. miss×2면 N+1턴 후 due인데 정답한 턴 자체가 N+1이라 즉시 pop 가능
- 수정 위치: `src/hooks/useRetryQueue.ts` (rescheduleAfterCorrect 또는 popDueOrNull) + `src/components/NoteGame.tsx` (turnCounterRef 갱신 시점)

### 0.2 Lv5+ 조표 음표 비율 부족 🐛
**사용자 검증**: "조표가 붙은 음표가 안 나와서 swipe 검증 자체가 막힘."

- 사용자 의도 (설계 §3.3.아): 6:4 / 4:6 / 7:3 / 3:7 비율로 자연음 vs 조표 음 + treble/bass 골고루
- 실제 동작: 조표 음표 거의 안 나옴
- 수정 위치: `src/components/NoteGame.tsx`의 `generateKeyBatch()` 안 `pickBalancedCount()` 또는 `accSlots` 계산 로직

### 0.3 카운트다운 후 첫 음표 버퍼링
**설계 §3.3.나 + 사용자 메모리**: "카운트다운 끝남 = 음표 표시 + 사운드 동시. 버퍼링 없어야 함. 특히 Lv7-3 3초 제한에서 첫 음표 답 시간 부족."

- 수정 방향: 카운트다운 종료 후 200~500ms 안정화 지연 추가, 또는 첫 음표만 grace period
- 수정 위치: `src/components/NoteGame.tsx`의 `handleCountdownComplete`

## 0-1. 설계-코드 갭에서 발견된 정책 결정 항목 🔴⭐

다음 세션에서 사용자가 결정 후 코드 통일 필요.

### 0-1.1 단계 Clear 기준 통일
**설계 §3.2.사.A vs 코드 `PASS_CRITERIA`**

| 항목 | 설계 | 코드 |
|---|---|---|
| 정답률 | 80~85% | 80% |
| 반응속도 | 타이머 30~40% 이내 | (조건 없음) |
| 수행 횟수 | 7~10번 이상 | 5회 이상 |
| 최대 연속 정답 | (설계엔 없음) | 5 이상 |

**결정 필요**:
- [ ] 정답률 임계 (80% vs 85%)
- [ ] 반응속도 통과 조건 추가 여부 + 비율 (30% vs 40%)
- [ ] 수행 횟수 (5 vs 7 vs 10)
- [ ] 최대 연속 정답 조건 유지 vs 삭제

### 0-1.2 Sublevel 3 stage 수
- 설계: 한 세션 = 3 STAGE
- 코드: Sub3 = 4 stage (3+5+7+7)
- 결정: 3 stage로 줄일지, 4 stage 유지(설계 갱신)할지

### 0-1.3 미가입자 권한
- 설계 §5.가: Lv 1~2 가능
- 코드 `canAccessSublevel(guest)`: Lv 1만
- 결정: 어느 쪽으로 갈지

### 0-1.4 N+2 단일 STAGE 음표 배치
**설계 §3.2.다 ※**: "음표가 하나 씩 3개가 배치되는 STAGE는 오선지 범위를 고려하여 3 → 5개로 수정하는 것이 나은지 판단 필요"

### 0-1.5 재도전 배지 삭제 ⭐
**설계 §3.3.자**: "현재 구현돼 있는 기능 중 N+2의 음표가 나올 때 상단에 보이는 '재도전' 마크도 삭제 필요. 무의미한 기능임."

- 수정 위치: `src/components/NoteGame.tsx`의 `🔁 재출제` 배지 부분
- 즉시 삭제 가능

### 0-1.6 같은 조표 연속 학습 (오답 多 시)
**설계 §3.3.바.B ※**: "혹은 해당 STAGE에서 틀린 음이 2~3개 이상일 경우 N+2 로직을 그대로 적용하여 다음 STAGE에서도 동일한 조표의 문제로 적용하는 방안"

- 결정: 구현 vs 미구현

## 2. 배치고사 + 랭크 시스템 🔴⭐
**24항목 #19, #20, #21, #22 + 배치고사 기획서 2개 + 설계 §4 종합**

### 2.1 신병 배치고사 (Initial Calibration) 🔴
- 가입 직후 튜토리얼 없이 60초 극한 테스트
- n+2 알고리즘으로 실시간 난이도 상승
- 3회 연속 오답 또는 반응속도 임계 초과 시 종료
- 결과로 초기 티어 배정

### 2.2 정규 배치고사 (Audition) 🔴
**자격 검정 (Qualification Phase)** — 응시 자격 자체가 리텐션 장치:
- 최근 3~5일 연속 연습 (Streak)
- 현재 레벨에서 음표 500개 이상 판독
- 최근 평균 정확도 90% 이상
- **Pro 구독자 혜택**: 자격 완화 (3일 → 1일)

**평가 산식**:
```
Score = (정답률 × 0.4) + (평균 반응속도 ms × 0.6)
```

### 2.3 티어 시스템 🔴
| 티어 | 세부 등급 | 시각 컨셉 |
|---|---|---|
| **Apprentice (수습)** | 3, 2, 1 | 미니멀 나무 업라이트 피아노 — 기초 음표 |
| **Performer (연주자)** | 3, 2, 1 | 블랙 유광 피아노 + 은빛 — 샵/플랫, 기본 화음 |
| **Virtuoso (거장)** | 3, 2, 1 | 골드 액센트 그랜드 — 복합 텐션, 빠른 템포 |
| **Maestro (마에스트로)** | 단일 (최상위 1%) | 크리스탈/오로라 그랜드 — 챌린지 굿즈 |

각 티어별 디자인 분리 — 레벨 오를수록 시각적 우월감 (24항목 #19)

### 2.4 랭크 감쇠 + 유지 🔴
- 주 3회 미만 연습 → 등급 포인트 하락
- 매월 1회 등급 유지 테스트 통과 필수
- **Pro 혜택**: 강등 보호권 + 스트릭 프리즈

### 2.5 배치고사 진행도 가시화 🟠
**24항목 #20**
- 화면에 "배치고사 응시 자격까지 X% 진행" 표시
- 일별 연속 연습 카운터, 누적 음표 수, 평균 정확도 대시보드
- 동기 부여 + 결제 전환 트리거

### 2.6 레벨 부여 기준 결정 🔴
**24항목 #22 + 설계 §4**
- 21단계 (Lv 1~7 × Sub 1~3) ↔ 티어 시스템 매핑 결정
- 예: Apprentice 3 = Lv 1-1 ~ Lv 1-3? 또는 별도 차원?
- **CTO 의견**: 레벨(콘텐츠 진도) ≠ 티어(실력 평가) 두 차원으로 분리하는 게 자연스러움

Green Billion 명세 명시: **"0.0001초(0.1ms) 정밀도 달성"**

> 이 섹션은 Green Billion Inc. "초정밀 타이밍 제어 공정 설계 명세서 v1.0" (2026-04-28)을 정식 기술 기준으로 채택한다.

### 7.1 고해상도 타임스탬프 표준화 🔴⭐
**Green Billion §2.1**

- **핵심 로직**: `Date.now()` 대신 브라우저의 `performance.now()` API를 단일 측정 기준으로 사용
- **목적**: OS·시스템 클록 의존 제거, 마이크로초(µs) 단위 상대 시간 측정 보장
- **구현 가이드**: 모든 입력 이벤트 발생 시 `performance.now()` 값을 캡처하여 로컬 변수에 즉시 할당

**현재 코드 위치 — 변경 필요**:
- `src/components/NoteGame.tsx`: `Date.now()` 호출 다수 (noteStartTime, recordNote reactionMs 등)
- `src/components/CountdownTimer.tsx`: startRef.current = Date.now() — 이미 박힘 → performance.now()로 전환
- `src/hooks/useNoteLogger.ts`: response_time 계산 부분
- `src/hooks/useSessionRecorder.ts`: recordNote의 reactionMs 측정

**검증 방법**: 입력 → 응답 사이의 측정 분산이 1ms 이하인지 확인

### 7.2 실행 스레드 분리 (Web Workers) 🔴⭐
**Green Billion §2.2**

- **핵심 로직**: 타이밍 측정 + 판정 알고리즘 연산을 메인 UI 스레드에서 분리해 **Web Worker**에서 실행
- **목적**: UI 렌더링 부하(Reflow, Repaint)로 인한 연산 지연·타이밍 오차 원천 차단
- **구현 가이드**: Worker ↔ 메인 스레드 메시징 오차 최소화. 최소 데이터만 직렬화

**구현 작업**:
- 새 파일: `src/workers/timingWorker.ts` 또는 유사
- 분리 대상: 정답 판정, 반응속도 계산, retry queue due 비교, mastery 가중치 계산
- 메시지 프로토콜: `{ type: "answer", note: ..., timestamp: ..., target: ... }` → `{ type: "result", correct: bool, reactionMs: number }`
- React 연동: `useEffect`에서 Worker 인스턴스 생성·소멸, `postMessage`로 통신

**난이도**: 높음. Vite + Web Worker 빌드 설정 + TypeScript Worker 타입 정의 필요.

### 7.3 사용자 환경 레이턴시 보정 (Calibration) 🔴⭐
**Green Billion §2.3**

- **핵심 로직**: 서비스 진입 시 사용자의 기기 성능·오디오 출력(블루투스 등) 레이턴시 측정 **캘리브레이션 루틴 강제**
- **목적**: 통제 불가능한 외부 변수(하드웨어 성능)를 수치화하여 보정값(Offset)으로 관리
- **구현 가이드**: 측정 오차 평균값을 `User_Environment_Offset`으로 정의, 실제 판정 시 `(실제 입력값 - Offset)` 공식 적용

**구현 작업**:
- 첫 게임 진입 시 캘리브레이션 모달 (또는 설정에서 수동 재실행)
- 측정 방법: 화면에 시각 신호 + 사운드 동시 제시 → 사용자 탭 → 입력 지연 측정 (3~5회 반복)
- `localStorage`에 `noteflex.userEnvOffset` 저장
- profiles 테이블에도 동기화 (디바이스 변경 시 재측정 권장)
- 측정값 적용: 모든 reactionMs 계산에서 `actualMs - userEnvOffset` 적용

**UI 디자인**:
- 캘리브레이션 모달 = "정밀 측정을 위해 1분만 협조 부탁드립니다" + 진행률
- 결과 표시: "측정 완료. 평균 입력 지연 23ms"

### 7.4 연산 무결성 확보 (Fixed-point Arithmetic) 🔴⭐
**Green Billion §2.4**

- **핵심 로직**: 모든 시간 연산은 부동소수점 오차 방지 위해 **정수(Integer) 기반 나노초/마이크로초 단위**로 처리
- **목적**: JavaScript 부동소수점 연산 특성으로 인한 0.0001초 단위 미세 계산 왜곡 방지
- **구현 가이드**: 연산 시 `100,000`을 곱하여 정수화 → 처리 → 최종 결과만 다시 소수점으로 변환하여 출력

**구현 작업**:
- 새 유틸: `src/lib/preciseTime.ts`
  - `toFixedInt(seconds: number): number` — × 100,000
  - `fromFixedInt(int: number): number` — ÷ 100,000
  - 또는 BigInt로 ns 단위 처리
- 적용 위치: 반응속도 비교, 평균 계산, 통과 기준 검증
- 테스트: 0.0001초 단위 입력에서 1:1 대조 통과 확인

### 7.5 유효 정밀도 정의 및 면책 (SLA) 🔴⭐
**Green Billion §2.5**

- **핵심 로직**: 기기 환경에 따른 **'유효 정밀도(Effective Precision)'** 지표 산출 → UI 표시
- **목적**: 소프트웨어 통제 불가 물리 한계 사용자 고지 → 기술 신뢰성 방어
- **구현 가이드**: 저사양 기기 또는 과도한 레이턴시 시 '정밀도 주의' 경고 노출 + 이용약관 §3 (기술 면책) 연동

**구현 작업**:
- `src/lib/effectivePrecision.ts`:
  - 입력: User_Environment_Offset, 디바이스 메모리, 네트워크 지연
  - 출력: "고정밀" / "표준" / "주의 (저사양)" 등급
- UI 위치: 게임 진입 전 또는 메인 페이지 우측 상단 작은 배지
- 등급별 처리:
  - 고정밀 (Offset < 30ms): 모든 기능 정상
  - 표준 (30~80ms): 정상이지만 배지 표시

- **이용약관 §3 신설 필요** — 기술적 면책 조항 추가
